ਸਭ ਸੰਭਵ ਹੈ ਕਿ ਤੁਸੀਂ ਪਹਿਲਾਂ ਜਾਣਕਾਰੀ ਸਟੋਰ ਕਰਨ ਲਈ ਸਪ੍ਰੈਡਸ਼ੀਟ ਦੀ ਵਰਤੋਂ ਕੀਤੀ ਹੋਵੇਗੀ। ਤੁਹਾਡੇ ਕੋਲ ਕਤਾਰਾਂ ਅਤੇ ਕਾਲਮਾਂ ਦਾ ਇੱਕ ਸੈੱਟ ਸੀ, ਜਿੱਥੇ ਕਤਾਰਾਂ ਵਿੱਚ ਜਾਣਕਾਰੀ (ਜਾਂ ਡਾਟਾ) ਹੁੰਦੀ ਸੀ, ਅਤੇ ਕਾਲਮਾਂ ਉਸ ਜਾਣਕਾਰੀ ਨੂੰ ਵਰਣਨ ਕਰਦੇ ਸਨ (ਕਈ ਵਾਰ ਇਸਨੂੰ ਮੈਟਾਡਾਟਾ ਕਿਹਾ ਜਾਂਦਾ ਹੈ)। ਇੱਕ ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਕਾਲਮਾਂ ਅਤੇ ਕਤਾਰਾਂ ਦੇ ਇਸ ਮੁੱਖ ਸਿਧਾਂਤ 'ਤੇ ਬਣਾਇਆ ਗਿਆ ਹੈ, ਜੋ ਤੁਹਾਨੂੰ ਜਾਣਕਾਰੀ ਨੂੰ ਕਈ ਟੇਬਲਾਂ ਵਿੱਚ ਫੈਲਾਉਣ ਦੀ ਆਗਿਆ ਦਿੰਦਾ ਹੈ। ਇਹ ਤੁਹਾਨੂੰ ਜਟਿਲ ਡਾਟਾ ਨਾਲ ਕੰਮ ਕਰਨ, ਡੁਪਲੀਕੇਸ਼ਨ ਤੋਂ ਬਚਣ ਅਤੇ ਡਾਟਾ ਦੀ ਜਾਂਚ ਕਰਨ ਦੇ ਤਰੀਕੇ ਵਿੱਚ ਲਚਕਦਾਰਤਾ ਰੱਖਣ ਦੀ ਆਗਿਆ ਦਿੰਦਾ ਹੈ। ਆਓ ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਦੇ ਸਿਧਾਂਤਾਂ ਦੀ ਜਾਂਚ ਕਰੀਏ।

## ਇਹ ਸਾਰਾ ਟੇਬਲਾਂ ਨਾਲ ਸ਼ੁਰੂ ਹੁੰਦਾ ਹੈ

ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਦਾ ਮੁੱਖ ਹਿੱਸਾ ਟੇਬਲ ਹੁੰਦੇ ਹਨ। ਜਿਵੇਂ ਸਪ੍ਰੈਡਸ਼ੀਟ ਦੇ ਨਾਲ, ਇੱਕ ਟੇਬਲ ਕਾਲਮਾਂ ਅਤੇ ਕਤਾਰਾਂ ਦਾ ਸੰਗ੍ਰਹਿ ਹੁੰਦਾ ਹੈ। ਕਤਾਰ ਵਿੱਚ ਉਹ ਡਾਟਾ ਜਾਂ ਜਾਣਕਾਰੀ ਹੁੰਦੀ ਹੈ ਜਿਸ ਨਾਲ ਅਸੀਂ ਕੰਮ ਕਰਨਾ ਚਾਹੁੰਦੇ ਹਾਂ, ਜਿਵੇਂ ਕਿ ਸ਼ਹਿਰ ਦਾ ਨਾਮ ਜਾਂ ਵਰਖਾ ਦੀ ਮਾਤਰਾ। ਕਾਲਮ ਉਸ ਡਾਟਾ ਨੂੰ ਵਰਣਨ ਕਰਦੇ ਹਨ ਜੋ ਉਹ ਸਟੋਰ ਕਰਦੇ ਹਨ।

ਆਓ ਆਪਣੀ ਜਾਂਚ ਸ਼ਹਿਰਾਂ ਬਾਰੇ ਜਾਣਕਾਰੀ ਸਟੋਰ ਕਰਨ ਲਈ ਇੱਕ ਟੇਬਲ ਸ਼ੁਰੂ ਕਰਕੇ ਕਰੀਏ। ਅਸੀਂ ਸ਼ਾਇਦ ਉਨ੍ਹਾਂ ਦੇ ਨਾਮ ਅਤੇ ਦੇਸ਼ ਨਾਲ ਸ਼ੁਰੂ ਕਰ ਸਕਦੇ ਹਾਂ। ਤੁਸੀਂ ਇਸਨੂੰ ਹੇਠਾਂ ਦਿੱਤੇ ਟੇਬਲ ਵਿੱਚ ਸਟੋਰ ਕਰ ਸਕਦੇ ਹੋ:

| -------- | ------------- |
| ਟੋਕਿਓ    | ਜਪਾਨ         |
| ਐਟਲਾਂਟਾ  | ਸੰਯੁਕਤ ਰਾਜ   |
| ਆਕਲੈਂਡ   | ਨਿਊਜ਼ੀਲੈਂਡ   |

ਧਿਆਨ ਦਿਓ ਕਿ **ਸ਼ਹਿਰ**, **ਦੇਸ਼** ਅਤੇ **ਆਬਾਦੀ** ਦੇ ਕਾਲਮ ਨਾਮ ਸਟੋਰ ਕੀਤੇ ਗਏ ਡਾਟਾ ਨੂੰ ਵਰਣਨ ਕਰਦੇ ਹਨ, ਅਤੇ ਹਰ ਕਤਾਰ ਵਿੱਚ ਇੱਕ ਸ਼ਹਿਰ ਬਾਰੇ ਜਾਣਕਾਰੀ ਹੁੰਦੀ ਹੈ।

## ਇੱਕ ਟੇਬਲ ਪਹੁੰਚ ਦੇ ਘਾਟ

ਸਭ ਸੰਭਵ ਹੈ, ਉਪਰੋਕਤ ਟੇਬਲ ਤੁਹਾਨੂੰ ਕਾਫ਼ੀ ਜਾਣੀ-ਪਛਾਣੀ ਲੱਗਦੀ ਹੈ। ਆਓ ਆਪਣੇ ਸ਼ੁਰੂਆਤੀ ਡਾਟਾਬੇਸ ਵਿੱਚ ਕੁਝ ਵਾਧੂ ਡਾਟਾ ਸ਼ਾਮਲ ਕਰੀਏ - ਸਾਲਾਨਾ ਵਰਖਾ (ਮਿਲੀਮੀਟਰ ਵਿੱਚ)। ਅਸੀਂ 2018, 2019 ਅਤੇ 2020 ਦੇ ਸਾਲਾਂ 'ਤੇ ਧਿਆਨ ਦੇਵਾਂਗੇ। ਜੇ ਅਸੀਂ ਇਹ ਟੋਕਿਓ ਲਈ ਸ਼ਾਮਲ ਕਰਨਾ ਹੈ, ਤਾਂ ਇਹ ਕੁਝ ਇਸ ਤਰ੍ਹਾਂ ਦੇਖਾਈ ਦੇ ਸਕਦਾ ਹੈ:

| ਸ਼ਹਿਰ  | ਦੇਸ਼    | ਸਾਲ | ਮਾਤਰਾ |
| ----- | ------- | ---- | ------ |
| ਟੋਕਿਓ | ਜਪਾਨ   | 2020 | 1690   |
| ਟੋਕਿਓ | ਜਪਾਨ   | 2019 | 1874   |
| ਟੋਕਿਓ | ਜਪਾਨ   | 2018 | 1445   |

ਤੁਹਾਨੂੰ ਸਾਡੇ ਟੇਬਲ ਬਾਰੇ ਕੀ ਦਿਖਾਈ ਦਿੰਦਾ ਹੈ? ਤੁਸੀਂ ਧਿਆਨ ਦੇ ਸਕਦੇ ਹੋ ਕਿ ਅਸੀਂ ਸ਼ਹਿਰ ਦਾ ਨਾਮ ਅਤੇ ਦੇਸ਼ ਨੂੰ ਵਾਰ-ਵਾਰ ਦੁਹਰਾ ਰਹੇ ਹਾਂ। ਇਹ ਕਾਫ਼ੀ ਸਟੋਰੇਜ ਲੈ ਸਕਦਾ ਹੈ, ਅਤੇ ਬਹੁਤ ਹੱਦ ਤੱਕ ਬੇਕਾਰ ਹੈ ਕਿ ਕਈ ਕਾਪੀਆਂ ਹੋਣ। ਆਖਰਕਾਰ, ਟੋਕਿਓ ਸਿਰਫ ਇੱਕ ਹੀ ਨਾਮ ਹੈ ਜਿਸ ਵਿੱਚ ਅਸੀਂ ਦਿਲਚਸਪੀ ਰੱਖਦੇ ਹਾਂ।

ਠੀਕ ਹੈ, ਆਓ ਕੁਝ ਹੋਰ ਕੋਸ਼ਿਸ਼ ਕਰੀਏ। ਆਓ ਹਰ ਸਾਲ ਲਈ ਨਵੇਂ ਕਾਲਮ ਸ਼ਾਮਲ ਕਰੀਏ:

| ਸ਼ਹਿਰ     | ਦੇਸ਼          | 2018 | 2019 | 2020 |
| -------- | ------------- | ---- | ---- | ---- |
| ਟੋਕਿਓ    | ਜਪਾਨ         | 1445 | 1874 | 1690 |
| ਐਟਲਾਂਟਾ  | ਸੰਯੁਕਤ ਰਾਜ   | 1779 | 1111 | 1683 |
| ਆਕਲੈਂਡ   | ਨਿਊਜ਼ੀਲੈਂਡ   | 1386 | 942  | 1176 |

ਜਦੋਂ ਕਿ ਇਹ ਕਤਾਰਾਂ ਦੇ ਦੁਹਰਾਅ ਤੋਂ ਬਚਦਾ ਹੈ, ਇਹ ਕੁਝ ਹੋਰ ਚੁਣੌਤੀਆਂ ਪੈਦਾ ਕਰਦਾ ਹੈ। ਸਾਨੂੰ ਹਰ ਵਾਰ ਨਵਾਂ ਸਾਲ ਹੋਣ 'ਤੇ ਆਪਣੇ ਟੇਬਲ ਦੀ ਬਣਤਰ ਨੂੰ ਸੋਧਣ ਦੀ ਲੋੜ ਹੋਵੇਗੀ। ਇਸ ਤੋਂ ਇਲਾਵਾ, ਜਦੋਂ ਸਾਡਾ ਡਾਟਾ ਵਧਦਾ ਹੈ ਤਾਂ ਸਾਡੇ ਸਾਲਾਂ ਨੂੰ ਕਾਲਮਾਂ ਵਜੋਂ ਰੱਖਣਾ ਮੁੱਲਾਂ ਨੂੰ ਪ੍ਰਾਪਤ ਕਰਨ ਅਤੇ ਗਣਨਾ ਕਰਨ ਨੂੰ ਔਖਾ ਬਣਾ ਦੇਵੇਗਾ।

ਇਸ ਲਈ ਸਾਨੂੰ ਕਈ ਟੇਬਲਾਂ ਅਤੇ ਰਿਸ਼ਤਿਆਂ ਦੀ ਲੋੜ ਹੈ। ਆਪਣੇ ਡਾਟਾ ਨੂੰ ਤੋੜ ਕੇ ਅਸੀਂ ਦੁਹਰਾਅ ਤੋਂ ਬਚ ਸਕਦੇ ਹਾਂ ਅਤੇ ਆਪਣੇ ਡਾਟਾ ਨਾਲ ਕੰਮ ਕਰਨ ਦੇ ਤਰੀਕੇ ਵਿੱਚ ਹੋਰ ਲਚਕਦਾਰਤਾ ਰੱਖ ਸਕਦੇ ਹਾਂ।

## ਰਿਸ਼ਤਿਆਂ ਦੇ ਸਿਧਾਂਤ

ਆਓ ਆਪਣੇ ਡਾਟਾ ਵੱਲ ਵਾਪਸ ਆਈਏ ਅਤੇ ਨਿਰਧਾਰਤ ਕਰੀਏ ਕਿ ਅਸੀਂ ਇਸਨੂੰ ਕਿਵੇਂ ਵੰਡਣਾ ਚਾਹੁੰਦੇ ਹਾਂ। ਅਸੀਂ ਜਾਣਦੇ ਹਾਂ ਕਿ ਅਸੀਂ ਆਪਣੇ ਸ਼ਹਿਰਾਂ ਲਈ ਨਾਮ ਅਤੇ ਦੇਸ਼ ਸਟੋਰ ਕਰਨਾ ਚਾਹੁੰਦੇ ਹਾਂ, ਇਸ ਲਈ ਇਹ ਸ਼ਾਇਦ ਇੱਕ ਟੇਬਲ ਵਿੱਚ ਸਭ ਤੋਂ ਵਧੀਆ ਕੰਮ ਕਰੇਗਾ।

| ਸ਼ਹਿਰ     | ਦੇਸ਼          |
| -------- | ------------- |
| ਟੋਕਿਓ    | ਜਪਾਨ         |
| ਐਟਲਾਂਟਾ  | ਸੰਯੁਕਤ ਰਾਜ   |
| ਆਕਲੈਂਡ   | ਨਿਊਜ਼ੀਲੈਂਡ   |

ਪਰ ਅਗਲੇ ਟੇਬਲ ਨੂੰ ਬਣਾਉਣ ਤੋਂ ਪਹਿਲਾਂ, ਸਾਨੂੰ ਇਹ ਪਤਾ ਲਗਾਉਣ ਦੀ ਲੋੜ ਹੈ ਕਿ ਹਰ ਸ਼ਹਿਰ ਨੂੰ ਕਿਵੇਂ ਰਿਫਰੈਂਸ ਕਰਨਾ ਹੈ। ਸਾਨੂੰ ਕਿਸੇ ਤਰ੍ਹਾਂ ਦਾ ਪਛਾਣਕਰਤਾ, ID ਜਾਂ (ਤਕਨੀਕੀ ਡਾਟਾਬੇਸ ਸ਼ਬਦਾਵਲੀ ਵਿੱਚ) ਇੱਕ ਪ੍ਰਾਇਮਰੀ ਕੀ ਦੀ ਲੋੜ ਹੈ। ਪ੍ਰਾਇਮਰੀ ਕੀ ਇੱਕ ਮੁੱਲ ਹੈ ਜੋ ਇੱਕ ਟੇਬਲ ਵਿੱਚ ਇੱਕ ਵਿਸ਼ੇਸ਼ ਕਤਾਰ ਦੀ ਪਛਾਣ ਕਰਨ ਲਈ ਵਰਤਿਆ ਜਾਂਦਾ ਹੈ। ਜਦੋਂ ਕਿ ਇਹ ਖੁਦ ਇੱਕ ਮੁੱਲ 'ਤੇ ਆਧਾਰਿਤ ਹੋ ਸਕਦਾ ਹੈ (ਉਦਾਹਰਣ ਲਈ, ਅਸੀਂ ਸ਼ਹਿਰ ਦਾ ਨਾਮ ਵਰਤ ਸਕਦੇ ਹਾਂ), ਇਹ ਲਗਭਗ ਹਮੇਸ਼ਾ ਇੱਕ ਨੰਬਰ ਜਾਂ ਹੋਰ ਪਛਾਣਕਰਤਾ ਹੋਣਾ ਚਾਹੀਦਾ ਹੈ। ਅਸੀਂ ਨਹੀਂ ਚਾਹੁੰਦੇ ਕਿ ID ਕਦੇ ਵੀ ਬਦਲੇ ਕਿਉਂਕਿ ਇਹ ਰਿਸ਼ਤੇ ਨੂੰ ਤੋੜ ਦੇਵੇਗਾ। ਤੁਸੀਂ ਜ਼ਿਆਦਾਤਰ ਮਾਮਲਿਆਂ ਵਿੱਚ ਪਾਓਗੇ ਕਿ ਪ੍ਰਾਇਮਰੀ ਕੀ ਜਾਂ ID ਇੱਕ ਆਟੋ-ਜਨਰੇਟ ਕੀਤੇ ਨੰਬਰ ਹੋਵੇਗਾ।

ਜਦੋਂ ਸਾਡਾ ਸ਼ਹਿਰਾਂ ਦਾ ਟੇਬਲ ਬਣਾਇਆ ਗਿਆ ਹੈ, ਆਓ ਵਰਖਾ ਸਟੋਰ ਕਰੀਏ। ਸ਼ਹਿਰ ਬਾਰੇ ਪੂਰੀ ਜਾਣਕਾਰੀ ਨੂੰ ਦੁਹਰਾਉਣ ਦੀ ਬਜਾਏ, ਅਸੀਂ ID ਦੀ ਵਰਤੋਂ ਕਰ ਸਕਦੇ ਹਾਂ। ਸਾਨੂੰ ਇਹ ਵੀ ਯਕੀਨੀ ਬਣਾਉਣਾ ਚਾਹੀਦਾ ਹੈ ਕਿ ਨਵੀਂ ਬਣਾਈ ਗਈ ਟੇਬਲ ਵਿੱਚ ਇੱਕ *id* ਕਾਲਮ ਵੀ ਹੈ, ਕਿਉਂਕਿ ਸਾਰੀਆਂ ਟੇਬਲਾਂ ਵਿੱਚ ਇੱਕ id ਜਾਂ ਪ੍ਰਾਇਮਰੀ ਕੀ ਹੋਣੀ ਚਾਹੀਦੀ ਹੈ।

ਨਵੀਂ ਬਣਾਈ ਗਈ **rainfall** ਟੇਬਲ ਦੇ ਅੰਦਰ **city_id** ਕਾਲਮ ਨੂੰ ਧਿਆਨ ਦਿਓ। ਇਸ ਕਾਲਮ ਵਿੱਚ ਉਹ ਮੁੱਲ ਹਨ ਜੋ **cities** ਟੇਬਲ ਵਿੱਚ ID ਨੂੰ ਰਿਫਰੈਂਸ ਕਰਦੇ ਹਨ। ਤਕਨੀਕੀ ਰਿਲੇਸ਼ਨਲ ਡਾਟਾ ਸ਼ਬਦਾਵਲੀ ਵਿੱਚ, ਇਸਨੂੰ **ਫਾਰਨ ਕੀ** ਕਿਹਾ ਜਾਂਦਾ ਹੈ; ਇਹ ਦੂਜੇ ਟੇਬਲ ਤੋਂ ਇੱਕ ਪ੍ਰਾਇਮਰੀ ਕੀ ਹੈ। ਤੁਸੀਂ ਇਸਨੂੰ ਸਿਰਫ ਇੱਕ ਰਿਫਰੈਂਸ ਜਾਂ ਪੌਇੰਟਰ ਵਜੋਂ ਸੋਚ ਸਕਦੇ ਹੋ। **city_id** 1 ਟੋਕਿਓ ਨੂੰ ਰਿਫਰੈਂਸ ਕਰਦਾ ਹੈ।

> [!NOTE] 
> ਫਾਰਨ ਕੀ ਨੂੰ ਅਕਸਰ FK ਵਜੋਂ ਸੰਖੇਪ ਕੀਤਾ ਜਾਂਦਾ ਹੈ

## ਡਾਟਾ ਪ੍ਰਾਪਤ ਕਰਨਾ

ਜਦੋਂ ਸਾਡਾ ਡਾਟਾ ਦੋ ਟੇਬਲਾਂ ਵਿੱਚ ਵੰਡਿਆ ਗਿਆ ਹੈ, ਤੁਸੀਂ ਸੋਚ ਰਹੇ ਹੋ ਸਕਦੇ ਹੋ ਕਿ ਅਸੀਂ ਇਸਨੂੰ ਕਿਵੇਂ ਪ੍ਰਾਪਤ ਕਰੀਏ। ਜੇ ਅਸੀਂ MySQL, SQL Server ਜਾਂ Oracle ਵਰਗੇ ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਦੀ ਵਰਤੋਂ ਕਰ ਰਹੇ ਹਾਂ, ਤਾਂ ਅਸੀਂ ਇੱਕ ਭਾਸ਼ਾ ਦੀ ਵਰਤੋਂ ਕਰ ਸਕਦੇ ਹਾਂ ਜਿਸਨੂੰ Structured Query Language ਜਾਂ SQL ਕਿਹਾ ਜਾਂਦਾ ਹੈ। SQL (ਕਈ ਵਾਰ sequel ਕਿਹਾ ਜਾਂਦਾ ਹੈ) ਇੱਕ ਮਿਆਰੀ ਭਾਸ਼ਾ ਹੈ ਜੋ ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਵਿੱਚ ਡਾਟਾ ਪ੍ਰਾਪਤ ਕਰਨ ਅਤੇ ਸੋਧਣ ਲਈ ਵਰਤੀ ਜਾਂਦੀ ਹੈ।

ਡਾਟਾ ਪ੍ਰਾਪਤ ਕਰਨ ਲਈ ਤੁਸੀਂ ਕਮਾਂਡ `SELECT` ਦੀ ਵਰਤੋਂ ਕਰਦੇ ਹੋ। ਇਸਦਾ ਮੁੱਖ ਅਰਥ ਹੈ ਕਿ ਤੁਸੀਂ **ਉਨ੍ਹਾਂ ਕਾਲਮਾਂ ਨੂੰ ਚੁਣੋ** ਜੋ ਤੁਸੀਂ ਦੇਖਣਾ ਚਾਹੁੰਦੇ ਹੋ **ਉਸ ਟੇਬਲ ਤੋਂ** ਜਿਸ ਵਿੱਚ ਉਹ ਸ਼ਾਮਲ ਹਨ। ਜੇ ਤੁਸੀਂ ਸਿਰਫ ਸ਼ਹਿਰਾਂ ਦੇ ਨਾਮ ਦਿਖਾਉਣਾ ਚਾਹੁੰਦੇ ਹੋ, ਤਾਂ ਤੁਸੀਂ ਹੇਠਾਂ ਦਿੱਤੇ ਕਮਾਂਡ ਦੀ ਵਰਤੋਂ ਕਰ ਸਕਦੇ ਹੋ:

```sql
SELECT city
FROM cities;

-- Output:
-- Tokyo
-- Atlanta
-- Auckland
```

`SELECT` ਜਿੱਥੇ ਤੁਸੀਂ ਕਾਲਮਾਂ ਦੀ ਸੂਚੀ ਦਿੰਦੇ ਹੋ, ਅਤੇ `FROM` ਜਿੱਥੇ ਤੁਸੀਂ ਟੇਬਲਾਂ ਦੀ ਸੂਚੀ ਦਿੰਦੇ ਹੋ।

> [!NOTE] 
> SQL ਸਿੰਟੈਕਸ ਕੇਸ-ਇੰਸੈਂਸਿਟਿਵ ਹੈ, ਜਿਸਦਾ ਅਰਥ ਹੈ ਕਿ `select` ਅਤੇ `SELECT` ਇੱਕੋ ਹੀ ਹਨ। ਹਾਲਾਂਕਿ, ਤੁਸੀਂ ਜਿਸ ਕਿਸਮ ਦੇ ਡਾਟਾਬੇਸ ਦੀ ਵਰਤੋਂ ਕਰ ਰਹੇ ਹੋ ਉਸ ਦੇ ਅਨੁਸਾਰ ਕਾਲਮ ਅਤੇ ਟੇਬਲ ਕੇਸ-ਸੈਂਸਿਟਿਵ ਹੋ ਸਕਦੇ ਹਨ। ਇਸ ਲਈ, ਇਹ ਇੱਕ ਵਧੀਆ ਅਭਿਆਸ ਹੈ ਕਿ ਹਮੇਸ਼ਾ ਪ੍ਰੋਗਰਾਮਿੰਗ ਵਿੱਚ ਹਰ ਚੀਜ਼ ਨੂੰ ਕੇਸ-ਸੈਂਸਿਟਿਵ ਸਮਝੋ। ਜਦੋਂ SQL ਕਵੈਰੀਆਂ ਲਿਖਦੇ ਹੋ, ਆਮ ਰਵਾਇਤ ਹੈ ਕਿ ਕੁੰਜੀਆਂ ਨੂੰ ਸਾਰੇ ਵੱਡੇ ਅੱਖਰਾਂ ਵਿੱਚ ਲਿਖੋ।

ਉਪਰੋਕਤ ਕਵੈਰੀ ਸਾਰੇ ਸ਼ਹਿਰਾਂ ਨੂੰ ਦਿਖਾਏਗੀ। ਆਓ ਕਲਪਨਾ ਕਰੀਏ ਕਿ ਅਸੀਂ ਸਿਰਫ ਨਿਊਜ਼ੀਲੈਂਡ ਵਿੱਚ ਸ਼ਹਿਰਾਂ ਨੂੰ ਦਿਖਾਉਣਾ ਚਾਹੁੰਦੇ ਹਾਂ। ਸਾਨੂੰ ਕਿਸੇ ਤਰ੍ਹਾਂ ਦੇ ਫਿਲਟਰ ਦੀ ਲੋੜ ਹੈ। SQL ਸ਼ਬਦ ਲਈ ਇਹ ਹੈ `WHERE`, ਜਾਂ "ਜਿੱਥੇ ਕੁਝ ਸੱਚ ਹੈ"।

```sql
SELECT city
FROM cities
WHERE country = 'New Zealand';

-- Output:
-- Auckland
```

## ਡਾਟਾ ਨੂੰ ਜੋੜਨਾ

ਅਜੇ ਤੱਕ ਅਸੀਂ ਸਿਰਫ ਇੱਕ ਟੇਬਲ ਤੋਂ ਡਾਟਾ ਪ੍ਰਾਪਤ ਕੀਤਾ ਹੈ। ਹੁਣ ਅਸੀਂ **cities** ਅਤੇ **rainfall** ਦੋਨੋਂ ਤੋਂ ਡਾਟਾ ਨੂੰ ਇਕੱਠਾ ਕਰਨਾ ਚਾਹੁੰਦੇ ਹਾਂ। ਇਹ ਦੋਨੋ ਨੂੰ *ਜੋੜਨ* ਦੁਆਰਾ ਕੀਤਾ ਜਾਂਦਾ ਹੈ। ਤੁਸੀਂ ਅਸਲ ਵਿੱਚ ਦੋ ਟੇਬਲਾਂ ਦੇ ਵਿਚਕਾਰ ਇੱਕ ਸਿਲਾਈ ਬਣਾਉਂਦੇ ਹੋ, ਅਤੇ ਹਰ ਟੇਬਲ ਤੋਂ ਇੱਕ ਕਾਲਮ ਦੇ ਮੁੱਲਾਂ ਨੂੰ ਮਿਲਾਉਂਦੇ ਹੋ।

ਸਾਡੇ ਉਦਾਹਰਣ ਵਿੱਚ, ਅਸੀਂ **city_id** ਕਾਲਮ ਨੂੰ **rainfall** ਵਿੱਚ **city_id** ਕਾਲਮ ਨਾਲ ਮਿਲਾਉਣ ਜਾ ਰਹੇ ਹਾਂ। ਇਹ ਵਰਖਾ ਦੀ ਮਾਤਰਾ ਨੂੰ ਇਸਦੇ ਸਬੰਧਤ ਸ਼ਹਿਰ ਨਾਲ ਮਿਲਾਉਣਗਾ। ਜੋੜ ਦੀ ਕਿਸਮ ਜੋ ਅਸੀਂ ਕਰਨ ਜਾ ਰਹੇ ਹਾਂ ਉਸਨੂੰ *inner* join ਕਿਹਾ ਜਾਂਦਾ ਹੈ, ਜਿਸਦਾ ਅਰਥ ਹੈ ਕਿ ਜੇ ਕੋਈ ਵੀ ਕਤਾਰਾਂ ਦੂਜੇ ਟੇਬਲ ਵਿੱਚ ਕੁਝ ਵੀ ਨਾਲ ਨਹੀਂ ਮਿਲਦੀਆਂ, ਤਾਂ ਉਹ ਦਿਖਾਈ ਨਹੀਂ ਦੇਣਗੀਆਂ। ਸਾਡੇ ਮਾਮਲੇ ਵਿੱਚ ਹਰ ਸ਼ਹਿਰ ਵਿੱਚ ਵਰਖਾ ਹੈ, ਇਸ ਲਈ ਸਭ ਕੁਝ ਦਿਖਾਈ ਦੇਵੇਗਾ।

ਆਓ 2019 ਲਈ ਸਾਰੇ ਸ਼ਹਿਰਾਂ ਲਈ ਵਰਖਾ ਪ੍ਰਾਪਤ ਕਰੀਏ।

ਅਸੀਂ ਇਹ ਕਦਮਾਂ ਵਿੱਚ ਕਰਨ ਜਾ ਰਹੇ ਹਾਂ। ਪਹਿਲਾ ਕਦਮ ਹੈ ਡਾਟਾ ਨੂੰ ਇਕੱਠਾ ਕਰਨਾ, ਦੋ ਟੇਬਲਾਂ ਨੂੰ **city_id** ਕਾਲਮ ਦੁਆਰਾ ਜੋੜਨਾ।

```sql
SELECT cities.city
    rainfall.amount
FROM cities
    INNER JOIN rainfall ON cities.city_id = rainfall.city_id
```

ਅਸੀਂ ਉਹ ਦੋ ਕਾਲਮ ਹਾਈਲਾਈਟ ਕੀਤੇ ਹਨ ਜੋ ਅਸੀਂ ਚਾਹੁੰਦੇ ਹਾਂ, ਅਤੇ ਇਹ ਗੱਲ ਕਿ ਅਸੀਂ **city_id** ਦੁਆਰਾ ਟੇਬਲਾਂ ਨੂੰ ਇਕੱਠਾ ਕਰਨਾ ਚਾਹੁੰਦੇ ਹਾਂ। ਹੁਣ ਅਸੀਂ ਸਿਰਫ 2019 ਸਾਲ ਨੂੰ ਫਿਲਟਰ ਕਰਨ ਲਈ `WHERE` ਸਟੇਟਮੈਂਟ ਸ਼ਾਮਲ ਕਰ ਸਕਦੇ ਹਾਂ।

```sql
SELECT cities.city
    rainfall.amount
FROM cities
    INNER JOIN rainfall ON cities.city_id = rainfall.city_id
WHERE rainfall.year = 2019

-- Output

-- city     | amount
-- -------- | ------
-- Tokyo    | 1874
-- Atlanta  | 1111
-- Auckland |  942
```

## ਸਾਰ

ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਕਈ ਟੇਬਲਾਂ ਵਿੱਚ ਜਾਣਕਾਰੀ ਨੂੰ ਵੰਡਣ 'ਤੇ ਕੇਂਦ੍ਰਿਤ ਹੁੰਦੇ ਹਨ, ਜੋ ਕਿ ਫਿਰ ਡਿਸਪਲੇਅ ਅਤੇ ਵਿਸ਼ਲੇਸ਼ਣ ਲਈ ਵਾਪਸ ਇਕੱਠੇ ਕੀਤੇ ਜਾਂਦੇ ਹਨ। ਇਹ ਗਣਨਾ ਕਰਨ ਅਤੇ ਹੋਰ ਤਰੀਕੇ ਨਾਲ ਡਾਟਾ ਨੂੰ ਮੋੜਨ ਲਈ ਇੱਕ ਉੱਚ ਡਿਗਰੀ ਦੀ ਲਚਕਦਾਰਤਾ ਪ੍ਰਦਾਨ ਕਰਦਾ ਹੈ। ਤੁਸੀਂ ਇੱਕ ਰਿਲੇਸ਼ਨਲ ਡਾਟਾਬੇਸ ਦੇ ਮੁੱਖ ਸਿਧਾਂਤਾਂ ਨੂੰ ਦੇਖਿਆ ਹੈ, ਅਤੇ ਦੋ ਟੇਬਲਾਂ ਦੇ ਵਿਚਕਾਰ ਇੱਕ ਜੋੜ ਕਰਨ ਦਾ ਤਰੀਕਾ ਸਿੱਖਿਆ ਹੈ।
